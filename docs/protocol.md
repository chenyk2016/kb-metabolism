# 知识代谢协议

> 权威版本。管理范围：`~/.openclaw/shared/notes/content` 下 00-inbox / 10-projects / 20-areas / 30-resources / 40-archive / 90-system 六区（2026-07-10 PARA 化迁移后）
> 核心原则：**知识库健康度不看有多少条，看每一条都有明确的未来用途。默认过期，主动续期。**

## 三层结构

```
L0 核心（人读）   ≤100 条硬上限。每条 = 一句可复述的判断 + 何时会再用。
L1 资料（AI 读）  能被检索到即可，人不通读。90 天零信号 → 处决名单。
inbox（暂存）     30 天到期。到期前要么升级（L0/L1），要么进名单。
L2 墓地           _graveyard/ 目录。随时可清空，git 保证可反悔。
```

## frontmatter 规范

每条被管理的笔记必须有：

```yaml
kb_id: aB3xK9pQ                            # 稳定身份，系统自动分配（出生/索引自愈），终生不变
kb_tier: L0 | L1 | inbox
kb_use_when: "什么时候会再用到（一句话）"   # L0/L1 必填
kb_triaged: 2026-07-07                     # 分诊日期
kb_expires: 2026-08-06                     # 仅 inbox 层，默认 +30 天
```

续命信号按 `kb_id` 认领而非路径——移动/重命名笔记不丢读取与引用记录，整理目录是安全动作。

**分诊铁律：写不出 `kb_use_when` 的内容，只能进 inbox 层等死。** 这是入口税。

## 分诊决策树（新内容进来时）

1. 是可执行任务？→ 不进知识库，进任务系统
2. 是一次性参考（这次用完就没用）？→ `inbox` + 30 天过期
3. 是长期知识？→ 能写出"什么时候会再用到"吗？
   - 能，且是判断/结论级 → `L0`（若已满 100 条，必须先挤掉一条）
   - 能，但是资料/细节级 → `L1`
   - 写不出 → `inbox`，等它自己证明价值或死掉

## 代谢节律

| 频率 | 动作 | 执行者 |
|---|---|---|
| 随时 | 检索必须走 kb-MCP（自动记访问日志） | 所有人和 agent |
| 每周 | `/kb-triage` 分诊新文件 + `/kb-weekly-digest` 消化 | agent 提案 |
| 每周 | 审批处决名单（勾选 → execute） | 人，5 分钟 |
| 每季 | 检查 L0 是否仍 ≤100，全库健康度 | 人 + agent |

## 处决判定（法医规则）

同时满足即进名单（AI 只提案，人批准后才执行）：
- 访问日志 90 天零读取
- 反向链接数 = 0（全 vault 范围扫描，daily 引用也算信号）
- git 最后修改 > 90 天
- inbox 层：`kb_expires` 已过即候选

执行 = `git mv` 到 `_graveyard/`，永远可反悔。墓地内容随时可真删。

## 废止条款

原《笔记记录规则》中「不擅自删除笔记（移动到 archive 代替）」对 00-my-inbox 范围废止，
替换为：**删除是默认路径，保留需要理由（use_when）；处决走名单审批，git 兜底。**
