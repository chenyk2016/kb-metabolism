KB-METABOLISM · 知识代谢系统 · 项目说明

让知识库学会遗忘

知识库的健康度不看存了多少条，看每一条都有明确的未来用途。默认过期，主动续期——文件为真相，一道带日志的门，定时作业提案，人当法官。

§ 01

病根：只有进，没有出

传统个人知识库的宿命：AI 时代收藏成本趋近于零，于是只进不出、越塞越多（全库已 2537 个文件），人无法通读，检索靠翻，删除靠勇气——最终变成一个没人敢清理的垃圾桶。

问题不出在存储或检索工具上，出在系统缺了两个器官：消化（把资料变成可复述的判断）和排泄（把失去用途的内容排出去）。本系统不是第 13 个工具，是给现有 Markdown 库装上这两个器官的代谢机制。

§ 02

代谢通路

捕捉 → 分诊 /kb-triage

新内容进来先定层写 frontmatter（kb_tier / kb_use_when），只加元数据，不动正文。

入口税：写不出「什么时候会再用到」的，只能进 inbox 等死

存活 → 使用即续命

所有检索走同一道门 kb_search / kb_read（MCP），每次访问自动记入日志——这是笔记「仍被使用」的唯一证据。

衰减 → 法医验尸 npm run coroner

每周扫一遍：90 天零读取、零反链、git 长期未动的笔记，进入处决名单；inbox 层 30 天到期即候选。

审判 → 人勾选 [x]

AI 只提案，人是法官。每周花 5 分钟勾选名单；不勾选即赦免，获得使用信号可自动脱榜。

排出 → 墓地 _graveyard/

npm run execute 对勾选项执行 git mv，索引同步重建。git 兜底，永远可反悔；墓地内容随时可真删。

§ 04

检索门：唯一入口，自动记账

MCP 工具作用代谢信号
kb_search混合检索：FTS5 中文 trigram + LIKE 兜底，返回路径 / 标题 / 摘要记录检索词
kb_read读笔记全文（仅限管理范围内）记录读取——续命的核心证据
kb_stats健康度：层级分布、L0 容量、孤儿笔记、近 7 天访问—

只需要记住一条纪律：查知识走 kb_search，别绕过门。
访问日志是全系统唯一不可再生的数据（重建索引也不清空）。信号从 2026-07-07 起算，你今天的每一次查询，都是三个月后法医手里的证据。

§ 05

法医判定与处决名单

L1 / 未分诊笔记进入名单，需三个信号同时归零：

  • AND访问日志 90 天零读取
  • AND反向链接数 = 0（全 vault 扫描，daily 笔记的引用也算活着）
  • ANDgit 最后修改超过 90 天

inbox 层规则更狠：kb_expires 一过即候选，不用等 90 天。前 90 天日志尚薄，判定以 git 时间 + 反链为主。

# 处决名单 2026-10-05 （示例）
  • - [x] `00-my-inbox/某次一次性调研.md` — [inbox] 已过期 35 天
  • - [ ] `00-my-inbox/风控改造步骤.md` — [L1] 0 反链；git 96 天未动 ← 人工赦免
$ npm run execute -- reports/kill-list-2026-10-05.md

§ 07

架构不变量

  • 真相层是纯 Markdown + git。SQLite、FTS 索引都是派生物，随时可删库重建；Obsidian 只是 L0 的阅读器。
  • 访问日志永不清空。它是唯一不可再生的代谢信号，重建索引时唯独保留 access_log 表。
  • 所有检索走同一道门。没有日志就没有衰减判定——绕过门的检索等于给全部笔记免死金牌。
  • 判断力按价值密度分配。分诊是高频低价值判断，用便宜模型；消化与 L0 挤位才值得顶级模型。
  • AI 提案，人判决。删除永远经名单勾选，git mv 保证每一次处决都可反悔。

§ 09

刻意不做

  • 不做向量检索。 39 条量级 FTS 足够；索引层可重建，日后加装无迁移成本。
  • 不做常驻 watcher。 分诊按需或每周批量跑，无状态 agent 干完即退。
  • 不自动开启 cron。 每次运行花模型调用费，开关在人手里。
  • 不做 Web UI。 Obsidian 就是 L0 的阅读器，本页就是说明书。
  • 不碰 00-my-inbox 以外的目录。 首期边界，跑通代谢闭环再谈扩张。
