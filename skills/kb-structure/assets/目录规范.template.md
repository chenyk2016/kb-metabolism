> 由 kb-structure 模板实例化，请按自己的库改写目标形态与当前状态；铁律、口诀、命名规范是经验证的默认值，删改前想清楚代价。

## 目标形态（顶层 10 项封顶，数字前缀锁定排序）

```
00-inbox       捕捉区：一切新笔记唯一入口（kb captureDir）
10-projects    进行中：有截止、有产出的事
20-areas       长期领域：工作、健康、家庭……（无终点）
30-resources   参考资料：wiki、文章存档、工具笔记
40-archive     唯一归档区：项目完结整体移入
90-system      规则、模板、MOC 索引笔记
daily          日记流（时间轴独立，可选）
_graveyard/ assets/   基础设施（下划线沉底，excluded）
```

> 有其他"租户"（多 agent 知识库、创作作品集）时，各占一个顶层编号位（如 50-agents、60-creative），总数仍不超 10；自治的租户区可加入上方 `exemptZones` 豁免深度/命名检查。

## 四条铁律

1. **顶层永不超 10 项。** 新目录必须回答"属于哪类行动性"（进行中/长期/参考/归档/他者/作品），答不出就不配存在。禁止主题目录——主题交给搜索、双链、MOC。
2. **深度 ≤2 层。** 需要第三层时改用 MOC 索引笔记（放 90-system）：一篇笔记可进多个 MOC，目录做不到。**归档区与墓地豁免**（把归档区加入 `exemptZones`）——深度限制服务于"人定位活跃内容"，进档保留原始结构。
3. **一个语义一个目录，一篇笔记一个路径。** 归档只有一处、墓地只有一处；禁止复制副本——精华用 L0 tier 或 MOC 表达，备份交给 git。
4. **捕捉零分类。** 新笔记一律落捕捉区（`日期-主题.md`），放哪里是分诊时的事，不是存的时候的事。

## 命名规范（每条规则附推导来源）

| 规则 | 为什么 |
|---|---|
| **basename 全库唯一**（不含扩展名的文件名不得重复） | kb 反链按 basename 解析——同名笔记会让反链记账错乱，这是实现机制的硬约束 |
| 捕捉笔记 `YYYY-MM-DD-主题.md` | 时间是捕捉时唯一零摩擦的轴；日期前缀天然排序、天然唯一化 |
| 目录 `NN-名称`（两位数字前缀） | 锁定侧边栏排序 + 可口头寻址（"放到 10 下面"） |
| 中文允许；禁止空格与 `\/:*?"<>\|`；连接用 `-` | 人可读优先；跨平台路径安全 |
| assets：每个顶层区一个 `assets/`，文件名用内容标识（token/hash） | 图片生命周期绑定笔记；区级粒度平衡同迁成本与路径深度；内容标识免撞名 |
| **改名 = 高危动作**：动 basename 即动 wikilink 解析，必须附全库引用改写 | 反链机制的另一面 |

## 安放口诀（分诊/整理时判定）

> 有截止 → 10；长期职责 → 20；只是参考 → 30；完事了 → 40；不是我的知识 → 其他租户位

## 移动纪律

- 移动/改名/重组永远先列提案（源 → 目标），人点头后执行——AI 提案，人是法官。
- 执行用 `git mv`；改名附全库引用改写；完成后跑 `kb index --vault <库根>`（kb_id 认亲，续命信号无损）。
- 移动带图笔记时，检查正文 `assets/` 相对引用，图片随笔记同迁。
- 删除类动作永不直接执行——转交 kb 处决名单流程。

## 体检节律

- 结构体检：`node <kb-structure 技能目录>/scripts/audit.mjs --vault <库根>`——六检报告即提案清单，建议随每周消化一起跑。
