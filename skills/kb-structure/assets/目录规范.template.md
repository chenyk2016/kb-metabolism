## 目标形态（顶层 10 项封顶，数字前缀锁定排序）

```
00-inbox       捕捉区：一切新笔记唯一入口（kb captureDir）
10-projects    进行中：有截止、有产出的事
20-areas       长期领域：工作、健康、家庭……（无终点）
30-resources   参考资料：wiki、文章存档、工具笔记
40-archive     唯一归档区：项目完结整体移入
90-system      规则、模板、MOC 索引笔记
daily          日记流（时间轴独立，可选）
_graveyard/ assets/   基础设施（下划线沉底，excluded）
```

> 有其他"租户"（多 agent 知识库、创作作品集）时，各占一个顶层编号位（如 50-agents、60-creative），总数仍不超 10。

## 四条铁律

1. **顶层永不超 10 项。** 新目录必须回答"属于哪类行动性"（进行中/长期/参考/归档/他者/作品），答不出就不配存在。禁止主题目录——主题交给搜索、双链、MOC。
2. **深度 ≤2 层。** 需要第三层时改用 MOC 索引笔记（放 90-system）：一篇笔记可进多个 MOC，目录做不到。
3. **一个语义一个目录，一篇笔记一个路径。** 归档只有一处、墓地只有一处；禁止复制副本——精华用 L0 tier 或 MOC 表达，备份交给 git。
4. **捕捉零分类。** 新笔记一律落捕捉区（`日期-主题.md`），放哪里是分诊时的事，不是存的时候的事。

## 安放口诀（分诊/整理时判定）

> 有截止 → 10；长期职责 → 20；只是参考 → 30；完事了 → 40；不是我的知识 → 其他租户位

## 移动纪律

- 移动/归档/重组永远先列提案（源 → 目标），人点头后执行——AI 提案，人是法官。
- 执行用 `git mv`；完成后跑 `kb index --vault <库根>`（kb_id 认亲，续命信号无损）。
- 移动带图笔记时，检查正文 `assets/` 相对引用，图片随笔记同迁。
- 归档动作顺手写元数据（tier 降级或归档标记）——kb 只读元数据，不懂目录。
