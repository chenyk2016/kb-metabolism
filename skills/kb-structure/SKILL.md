# kb-structure：目录结构守门

**先读宪法**：库内 `90-system/目录规范.md` 是唯一真相源（顶层形态、铁律、口诀、当前状态都在那里）。本技能只带执行程序；两者冲突时以库内文档为准——人改规则只改那个文件，本技能自动跟随。

宪法放库内而非本技能目录，是因为它是**库的可变状态**而非技能的知识：人要在 Obsidian 里看到它，其他工具也要读它，它的修订和库结构变更属于同一个 commit（vault 自包含：拷走目录 = 带走整个系统，包括宪法）。

**库里还没有宪法时**：向用户提议从本技能 `assets/目录规范.template.md` 实例化到库的 `90-system/`（模板是推荐默认，逐节和用户确认后写入）。用户不想建就只做最保守的动作（新笔记落 kb captureDir），不做任何移动或目录创建——不要凭记忆虚构规则。

## 为什么需要守门

这类库的检索归 kb_search、生死归 tier 信号，目录唯一的客户是人。目录的失败模式只有两种：顶层膨胀（几十项时人已无法扫视）和主题分类树（会交叉、会漂移、必然腐烂）。守门的全部意义是在"动手那一刻"拦住这两件事——事后整理的成本是当场拦截的百倍。

## 决策程序

**1. 新笔记要保存**
落宪法"当前状态"节指定的捕捉区，文件名 `YYYY-MM-DD-主题.md`，不做任何分类决策（捕捉零分类：分类摩擦会杀死捕捉欲）。属于 kb managed 范围的知识优先走 kb_add（它自带入口税与查重税）；本技能只在用户明确指定语义位置（"放到 XX 项目下"）时按口诀核对落点并直接执行。

**2. 要创建目录**
- 顶层新目录：默认拒绝。解释顶层封顶的原因，给出现有各项中的归属建议。用户坚持则照办——人是法官，但要先听到理由。
- 第 3 层及更深：不建目录，改为在 90-system 建/更新一篇 MOC 索引笔记（一篇笔记可进多个 MOC，目录做不到）。
- 2 层以内：过一道行动性问题——"这是进行中/长期/参考/归档哪一类？"答得出才建。

**3. 要移动/重命名/归档/整理**
先列提案清单（源 → 目标，附口诀依据），等用户点头再动。执行流程：
```
git mv <源> <目标>
kb index --vault <库根>     # kb_id 认亲，续命信号无损
```
带图笔记检查正文 `assets/` 相对引用，图片随笔记同迁。归档到归档区时顺手写元数据标记。批量整理分批提交，每批一个 commit。

**4. 要复制一篇笔记到别处**
拒绝，并按用途给替代：想多处可见 → 双链或 MOC；想标记重要 → 提 tier（走 kb 分诊）；想备份 → git 已经在备份。一篇笔记一个路径，副本会分裂续命信号、导致编辑分叉。

## 边界

- 移动、删除、顶层建目录永不自作主张——提案给人判决。
- 不改 `.kb/config.json`、不碰 tier/信号/处决名单（kb-metabolism 主技能的领地；分诊定层用它的 `kb triage`）。
- 只管装有 kb-metabolism 的库；库外目录有各自的规矩。
